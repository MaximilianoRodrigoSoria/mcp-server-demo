# Guion del podcast — mcp-server-demo

Segundo episodio del portfolio, y toca un tema que se volvió central en el mundo de los agentes de inteligencia artificial: el Model Context Protocol, o MCP. El proyecto se llama mcp-server-demo, y la idea es construir un servidor MCP que exponga herramientas propias para que un modelo de lenguaje las pueda usar de forma estandarizada.

Empecemos por entender qué problema resuelve MCP. Los modelos de lenguaje, por potentes que sean, están encerrados en su propio conocimiento. Para que hagan algo útil con tus datos o tus sistemas, necesitan herramientas: consultar una base de datos, llamar a una API, leer un archivo. Hasta hace poco, cada framework tenía su propia manera de definir esas herramientas, así que lo que construías para uno no servía para otro. MCP es un protocolo abierto que estandariza esa comunicación. Construís el servidor una vez, y lo puede consumir cualquier cliente compatible: Claude Desktop, un IDE, o el propio sistema multi-agente de este portfolio.

¿Qué expone concretamente nuestro servidor? Tres tipos de cosas. Primero, las tools, que son las acciones invocables. Vamos a exponer al menos dos: una que consulta una base de datos local en SQLite, y otra que llama a una API externa, por ejemplo una API pública de clima o de tipo de cambio. Segundo, los resources, que son datos que el modelo puede leer, como el esquema de la base de datos, para que sepa qué puede consultar. Y tercero, opcionalmente, los prompts, que son plantillas reutilizables y parametrizadas.

En cuanto al stack, hay dos caminos. La opción en Python usa el SDK oficial, mcp, con la clase FastMCP, y es la más rápida para desarrollar. La opción en TypeScript usa el SDK correspondiente, y sirve si querés mostrar versatilidad. Para el transporte hay dos modos: standard input output, o stdio, ideal para uso local con clientes de escritorio; y Streamable HTTP, para exponer el servidor de forma remota. La base de datos es SQLite con datos de ejemplo sembrados, y la API externa conviene que sea una pública y sin fricción, como Open-Meteo. La validación de entradas se hace con Pydantic en Python o Zod en TypeScript, y para probarlo tenés el MCP Inspector.

La estructura de carpetas separa con claridad las responsabilidades. Hay una carpeta de datos con el esquema y el script para sembrar la base. En el código fuente, un archivo de servidor que instancia todo, una carpeta de tools dividida entre las de base de datos y las de API externa, una carpeta de resources, y una capa de conexión a la base con queries parametrizadas. Ese detalle de las queries parametrizadas es importante: es lo que te protege de inyección de SQL, y es una de esas cosas que un revisor técnico nota enseguida.

Vamos al checklist, porque marca las prioridades. Después del setup y de sembrar la base, implementás las tools de base de datos, siempre con entradas tipadas y descripciones claras, porque el modelo decide cuándo usar cada herramienta justamente leyendo esa descripción. Después las tools de API externa, con timeout y reintentos en el cliente HTTP, y normalizando la respuesta a algo compacto y útil. Luego exponés el esquema como resource. Y hay una fase entera dedicada a robustez y seguridad: validar todas las entradas, no exponer secretos en respuestas ni en logs, leer las claves desde el entorno, y loguear las invocaciones con su latencia.

La fase de integración es la que cierra el proyecto y lo hace tangible. Probás el servidor con el MCP Inspector, listando y ejecutando las herramientas. Después lo registrás en Claude Desktop con un archivo de configuración de ejemplo y hacés una prueba real, con un modelo invocando tus tools contra la base y la API. Y como frutilla del postre, lo podés consumir desde el proyecto de orquestación multi-agente, conectando dos piezas del portfolio.

El criterio de terminado: un cliente MCP descubre las herramientas, un modelo las invoca correctamente contra la base de datos y la API externa, las entradas están validadas y los errores manejados, y funciona tanto en modo local por stdio como, opcionalmente, en modo remoto por HTTP. Con eso demostrás que sabés extender las capacidades de un modelo usando el estándar que está definiendo el rubro. Nos vemos en el próximo episodio.
